# 驾驭工程

软件工程师逐渐从"代码的编写者"变成"环境的建筑师"。最大的工程挑战是设计让 Agent 可靠工作的控制系统。

参考：
- [驾驭工程](https://www.runoob.com/ai-agent/harness-engineering.html)

**组件架构**：

- **上下文工程**（Context Engineering）——新员工手册
	比如 **Agent.md** 是 AI 智能体进入代码仓库时看到的第一份指南，通过这个文件了解整个项目的架构。
	为了尽可能少地占用上下文有限的空间资源，从 Agent.md 开始会多层级设置指南，且让 LLM 按需加载，典型的实现就是 **SDD** 以及 **Skills**；
	看 `openclaw` 源码会发现虽然没有明确说使用 SDD（OpenSpec、Spec-Kit）但是**基本每个模块(包括微小模块)都有详细的 MD 文档说明**。
	让 Agent 帮忙解析源码实现时，发现 Agent 也会借助这些文档定位和理解目标代码。

- **架构约束**（Architecture Constraints）——缰绳
	自定义 **Linter** 规则，这些规则是强制实施且明确的，且是**脚本触发**（触发逻辑是固化到 Agent 内部的），不会出现当 Agent 上下文过大时会忘记一些重要约束的问题。

- **反馈循环**（Feedback Loop）——智能体审智能体
	传统开发中，人类工程师负责代码审查（Code Review）。在驾驭工程中，这个工作变成了"**智能体对智能体**"的方式：Codex 在本地审核自身更改，请求额外审查，循环往复直到通过。
	反馈循环中的钩子可以运行预定义的测试套件，并在失败时带着错误信息循环回到模型，或者提示模型独立评估其代码。如果 AI 写的测试用例"通过"了带有 Bug 的代码，Harness 就会判定测试无效，强迫它重新思考测试边界。

- **熵管理**（Entropy Management）——垃圾回收
	定期清理项目迭代过程中积累的技术债务。
	具体措施：定期运行后台 Codex 任务扫描偏差、更新质量等级、发起针对性重构 PR。此外还有一个专门的 **Doc-gardening Agent**（文档园丁代理），在后台自动扫描文档与代码之间的不一致，发现过时内容就自动提交 PR 修复——Agent 为 Agent 维护文档。
