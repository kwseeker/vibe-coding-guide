# 规范驱动开发

## SDD 简介

**氛围编程的问题**：
面对复杂、长期维护或对可靠性要求极高的系统时，往往暴露出一些严重缺陷：
+ **上下文漂移**
	因空间限制导致的上下文遗忘与理解歧义问题。
+ **架构违约**
	也和Token上下文空间限制有关。
+ **需求一致性缺失**
	也和Token上下文空间限制有关。
+ **性能问题**
+ **安全漏洞**
+ **代码质量问题**
+ **忽略边界条件**

**规范开发的意义**：
规范驱动开发的出现就是为了解决上述问题。

**规范驱动开发的流程**：
需求 → 详细规范 → AI 生成 → 验证。
先规范、后代码；AI 根据规范实现，开发者聚焦**架构、需求与验证**；质量通过**系统化闸门**把关；并通过**持续反馈把错误信息融入规范**，迭代提升输出质量。
> 注意：规范不是一次就写完善的，规范文档也需要随着需求迭代，发现解决总结问题后不断进行完善。

## 维护规范文档

**规范文档的组成**：
- 目标与价值（解决什么问题）
- 上下文与约束（架构、依赖、环境、性能要求）
- 功能需求（核心行为与特性）
- 非功能需求（安全、性能、可扩展、可访问性）
- 边界与错误处理
- 测试标准
- 示例（输入/输出、样例数据、使用场景）

**规范的书写要求**：
- 清晰：无歧义，避免误解；
- 完整：边界与约束明确；
- 上下文充分：让 AI 理解领域与架构背景；
- 具体：用实例胜过抽象描述；
- 可测：定义清晰的验证标准。
