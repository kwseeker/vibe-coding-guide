# 上下文空间限制的应对策略

上下文超限导致模型变傻的原因：
+ 模型失忆
	Token 超限导致信息被裁剪
+ 注意力稀释（attention dilution）
	即便没有超限也会对历史注意力下降。
- 长链推理退化

## 主流编程 Agent 内置策略

+ **上下文压缩/内容摘要**（Conversation Summarization）
	对历史对话进行压缩，剔除细节。
	> 后续启动一个相关任务会自动取回细节么？
+ **代码检索**（Code Retrieval / RAG）与 **文件按需读取**（Lazy File Loading）
	不会加载整个项目到上下文，会根据语义进行文件检索，只加载相关文件。
- **任务分解**（Task Decomposition）
	将复杂任务拆分为多个小任务。
- **工具化外部记忆**（External Memory）
	像规范文件就属于外部存储。

**Gemini 3.1 Pro 的策略**：

> 问：你针对上下文空间限制问题有哪些应对策略？

Gemini 3.1 Pro 的回复：
为了打破“模型失忆”和“上下文爆炸”的限制，系统在底层设计上采取了多套极具针对性的应对策略：

-  **动态按需加载**（On-Demand Search & Retrieval）

	我**绝不会一次性读取整个代码库**，那会瞬间撑爆上下文。取而代之的是，拥有类似于“外科手术刀”的检索能力：
	- **AST 级别的解析**：我不必从头到尾读完一个几千行的文件。使用 `view_file_outline` 或者 `view_code_item`，我可以只拉取这个文件里的类名、函数签名和骨架架构。
	- **精准的分块阅读**：通过 `grep_search` 或指定行号（`StartLine`/`EndLine`），我可以像翻字典一样直击错误代码所处的上下几行，而不是加载整个文件。

- **局部代码流修改**（Targeted Chunk Edits）

	当需要修改代码时，我不需要把改好后的两千行文件全部在聊天框里向你输出一遍。
	- 我会使用 `replace_file_content` 或 `multi_replace_file_content`，仅在底层精准对齐目标块（TargetContent），就像用 Git Patch 一样打补丁。这极大地节省了系统往返的 Token 消耗，使我能一直保持思考的纯粹度。

- **长效记忆与知识提炼**（Persistent Context & KIs）

	每次你关闭并新开一个会话窗口，我们的长篇历史对话不会被全部硬塞入新会话中。
	- **KI 机制（Knowledge Items）**：在后端，有一个独立的知识抓取子 Agent 会静默分析我们过去的对话，并将特定的核心概念、解决方案提炼沉淀为高纯度的“知识项”。
	- 当新会话开始时，你的问题一旦触及之前的主题，我会从摘要列表里“顺藤摸瓜”，专门去提取与该主题相关的 KI 文档进行回顾。这种设计让我的大脑既能不断成长，又不会被流水账对话给填满。

- **产物隔离**（Artifacts System）

	- 当面对诸如“帮我写一份详细的组件设计方案”、“画个架构时序图”等高密度内容生成时，我不会选择把几千字全部输出在对话流里。
	- 我会使用 **Artifacts（生成产物记录档）**，直接在项目内的 `.gemini/antigravity/brain/...生成专门的结构化 Markdown 或报告文档`（包含代码差异、表格、图表），这样不但展示直观，也能够防止我们的主聊天链路因为冗余信息失去核心焦点。

- **限制高噪节点**（Terminal Output Truncation）

	- 在替你执行 `run_command`（如跑测试用例、拉取 npm 模块或打印各种海量日志错误）时，我的指令调度系统会主动截断超出必要分析长度的输出流（限制 CharacterCount 或者采用短时长的 Pager 工具）。我会像人类开发者一样寻找关键报错的栈底，而非去阅读滚屏了三十分钟的 Log 瀑布。
