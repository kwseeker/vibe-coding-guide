# Vibe Coding

## 氛围编程的问题

+ 缺乏主动性，当没有严谨的规范约束时，可能会缺失必要的错误处理、边界情况覆盖、安全漏洞检查、忽略可拓展性、引入性能问题
+ 不会自动优化、重构代码、复用代码

> 这些问题都需要人提前约定规范以及审查AI编写的代码来避免。
> 部署到生产环境的代码必须被人充分理解与验证。

## 如何配合 Agent 更好地工作

> 问：我应该怎么配合你来避免出现模型失忆和推理退化问题？我需要遵循哪些规范才能让你更好的工作？针对我以往的行为，有哪些需要改进的操作？

Gemini 3.1 Pro 针对历史习惯提供的建议：

### 配合我的 4 项核心规范

#### 1. 单点职责原则（SRP）：每次会话关注一个原子任务

- **为何这样做**：任何 LLM Agent 随着对话轮次的增加，其指令遵循能力都会呈指数级下降。如果在同一个会话里既重构了首页 UI，又去修了后端的 Postgres 数据库查询 bug，最后还让我写个文档，我的注意力机制极容易“串台”。
- **你应该**：修完某个小阶段的 Feature（比如完成了 M1），将代码 commit。如果下一个阶段 M2 跨度较大，**果断开启一个新会话（New Chat/Reset）**，向我发起：`@/openspec-apply implement-community-biz 继续实现 M2`。重启会话就等于清理了我的显存碎片。

#### 2. “文件”即“大脑”：少用大段废话，多用规范约束

- **为何这样做**：你在聊天框里敲的三千字需求，我会因为后面的对话慢慢将其权重降低（即“失忆”）。但如果你把这三千字写进了 `AGENTS.md` 或 `openspec/XXX/spec.md` 中，其作为工程级约束的权重将始终保持满级。
- **你应该**：把代码重构原则、UI 设计规范等统一收敛进工程特定目录文档中。你在提问时只需说“参考 @[docs/uiux/STYLE.md] 实现此页面”。

#### 3. 精确的“探照灯”锚定代码（@文件+行号）

- **为何这样做**：如果只说“首页显示不全”，我就要满世界地找首页入口、找路由、找组件，这个过程中极大消耗上下文。
- **你应该**：像你之前做的那样，提供高度精确的坐标，例如 `@[weapp/src/pages/recipe/index.tsx:L123-L456] 这个地方的布局溢出了`。使用精准 `@` 能让我瞬间绕开检索步骤，直接展开推理和修改。

#### 4. 不要隐式依赖时间线，要显式提供上下文状态

- 遇到报错或重构失败时，“刚才的方法不行”、“怎么报错了” 这类话容易让我陷入迷茫。
- 请用结构化的反馈语：`执行刚才的脚本 X 后，终端输出了报错 Y，我的目标是继续实现 Z，请分析报错并重试。`

### 💡 针对你过往行为的 3 点“可改进建议”

纵观我们最近这几十轮的交流，我发现你的实操习惯极好（特别是高频使用精确的文件引用和 OpenSpec 工具）。但依然有几个可以优化的地方，能让我的准确率更上一层楼：

**改进一：避免在一句话中同时下达多个层级截然不同的指令组**

- **以往回顾**：在 `2026-03-02 11:33:12` 你的 prompt 中，你同时要求我：`1. 将食材用量成比例修改逻辑补全。 2. 烹饪技巧这块设计的有点丑，借助 ui-ux-max-pro 技能进行优化... 考虑底部加入颜色等`。
- **改进建言**：这里包含了“复杂后端/数据驱动的逻辑算法”和“前端 UI 发散式设计重构”。这两个任务属于左右脑的不同模式，如果放在一轮，我可能会因为优化 UI 而将计算逻辑写出 bug。
- **操作推荐**：拆分 prompt。“先帮我把食材根据份量成比例修改的逻辑走通。”（确认无误后，再发一轮）“现在的烹饪技巧 UI 太丑，请运用 ui-ux-max-pro 技能进行重新设计。”

**改进二：依赖报错截断或指定进程 ID，不要让我猜错误在哪**

- **以往回顾**：在 `2026-03-01 20:13:25`，你说 `@[TerminalName: node, ProcessId: 28542] 这里为何会报 invalid code`。你能直接引用运行环境，这很好！
- **改进建言**：有时候终端的日志滚动极快。如果你能直接把最关键的 `error stack trace` (前 10 行报错) 顺手贴在聊天框，我就不用通过执行指令、读取终端日志的工具往返试探，这能为你节省数十秒的推演时间。

**改进三：关于新分支 / 新规范的“冷启动”唤醒**

- **以往回顾**：在 `2026-03-07 00:17:32` 你开启窗口直接说：`@[/openspec-apply]implement-community-biz 继续实现其他阶段任务`。
- **改进建言**：正如你问到的“会不会每次读 AGENTS.md”，答案是不会默认全扫。如果这个新会话没有直接包含关于我们进度到了哪一步的先验知识，我可能需要先去翻找 spec 文档才能知道“其他阶段任务”是指什么。
- **操作推荐**：在新窗口重启长链路任务时，带上一句状态锚点：`@[/openspec-apply]implement-community-biz 我们已经做完了 M1，现在请阅读相关提案并继续实现 M2 的前置工作。`这种显式的路标指引能让我的“冷启动”变得无比平滑。

总之，**把我当作一个拥有顶级编码能力、但“睡一觉就会忘记昨天开会细节”的资深外包工程专家**。每次派活儿时，给他看唯一的、明确的、结构化的《需求文档》，并给予精确的起始代码坐标，这就是你驯服、驾驭 Agent 系统并发挥其 100% 威力的黄金准则！


### 协作工程化手段

> 问：除了基于 规范驱动开发（比如 openspec）还有技能（Skill），还有没有其他方式可以优化我们协作的体验

Gemini 3.1 Pro 回复：

除了你当前使用得炉火纯青的 **OpenSpec（规范驱动开发）** 和 **Skills（技能包）**（比如你用来做设计的 `ui-ux-pro-max`），我们还可以通过以下 **3 种高阶工程化手段** 进一步优化协作体验。这些方式旨在降低你的管理成本，进一步释放我作为全栈 Agent 的自主潜能：
#### 1. 深度整合 MCP（Model Context Protocol）生态

你可能注意到了系统初始化时有 `<mcp_servers>` 这一环节（比如 `firebase-mcp-server`）。MCP 是一种能让 AI 安全、稳定地“触碰”外部数据源的标准协议。

- **你可以做什么**：
    1. 如果你的项目集成了外部服务（诸如 `PostgreSQL数据库`、`GitHub PR/Issue`、或是 `Figma 设计稿`），你可以通过配置对应的 MCP Server 将它们接入我的环境。
    2. 例如配置了 **SQL MCP**：你可以直接对我说：“帮我查一下 `users` 表里昨天注册、但没有发布任何食谱的用户数据”，我就可以自主构筑 SQL 语句查出在线数据并返还分析结果。
- **优化效果**：打破了只能帮你和“本地代码文本”打交道的局限，我可以帮你在终端中对真实的外部架构环境进行 DevOps 测试或业务数据核验。

#### 2. 沉淀与触发自定义 Workflows（工作流）

当前你使用的 `/openspec-apply` 等斜杠命令就是一个典型的基于 `Workflows` 的实现。

- **你可以做什么**： 你可以将高频且繁杂的多步骤命令转化为本地特定目录下的 YAML/Markdown 工作流文档（如 `.agent/workflows/[name].md`）。 例如：**`/deploy-staging`** 在这个文件内写明具体步骤：
    1. 跑 `pnpm test`
    2. 执行 `pnpm build`
    3. `git commit` 并把分支推送到 `origin staging`。 使用 `// turbo` （某一步）或 `// turbo-all` 标签。
- **优化效果**：当你下达 `/deploy-staging` 命令时，因为有 `turbo` 标签，我不再需要针对这繁琐的三部曲频繁地弹窗询问你的授权（“你要不要我运行这个？”），而是进入静默极速模式，像脚本一样一气呵成帮你扫除周边所有的 CI/CD 和环境部署工作。

#### 3. 多模态协作：录屏排障与视觉设计（Browser Subagent & Vision）

我们在 `Precise-Recipes` 中进行了大量的前端页面和 UIUX 优化工作。然而，很多时候你只能通过代码或者口头转述：“首页这个卡片尺寸被刘海遮挡了”。

- **你可以做什么**： 充分利用我的浏览器代理系统（`browser_subagent`）和图像生成编辑工具（`generate_image`）。 如果遇到了特定疑难的重排界面错位，或者某个前端库报错：
    - 你可以让我**自主打开一个本地测试链接的主流浏览器环境录制/截图**：“打开 `http://localhost:3000` 并帮我模拟一次发布食谱被阻断的流程，去看看浏览器 Console 里输出了什么错误。”
    - 我会在后台创建浏览器录制、截图并直接读取 DOM 结构给你一个非常立体的故障定位图。
- **优化效果**：这把我们的交互从单纯的“瞎子摸象式代码检视”，上升到了真正能看见效果图、能实机联调的真机预览段位。

**总结**，以你现在的体系，如果要在现有 `OpenSpec + Skills` 地基上再上一个台阶：

1. **向外破圈（MCP）** -> 让大模型连接你的 Figma 和数据库。
2. **向内放权（Workflows + Turbo）** -> 把重复的测试、部署流水线命令自动化掉，不用再向我频频授权。
3. **视觉直观（Browser Subagent）** -> 让我能够“看见”小程序/H5 真正渲染到屏幕上长什么样。
