# 驾驭工程

软件工程师逐渐从"代码的编写者"变成"环境的建筑师"。最大的工程挑战是设计让 Agent 可靠工作的控制系统。

## 工程思想发展

- **提示词工程**
	（定义如何描述清楚一个问题）
	- 澄清任务
	- 约束输出
	- 激发模型已有能力
- **上下文工程**
	（定义上下文的供给机制，确保按需精确供给合适的提示词）
	- 用户输入
	- 历史对话
	- 检索结果
	- 工具返回
	- 当前任务状态
	- 中间产物
	- 系统规则
	- 安全约束
- **驾驭工程**
	（约束模型按正确的方向执行且可以自行纠偏）
	（LangChain 工程师将 Agent 除去模型剩余的部分都算进 Harness）
	-  **信息边界**
		- 角色和目标定义
		- 信息选择和裁剪
			（上下文不是越多越好，而是越相关越好）
		- 结构化组织
			（将提示词（比如固定规则、任务列表、运行状态、外部数据）按分层结构存放并交代清楚，方便LLM按需检索）
			（结构化的提示词可以防止模型漏重点、忘记约束和自我污染。）
	- **工具系统**
		（接触真实世界的窗口）
		- 合理提供工具
			（研究人员发现工具给的太少能力不够，工具给的太多模型又会乱用）
		- 规定什么时候该调什么工具
		- 结果怎么重新喂回
			工具获取的结果不应该直接返回，需要根据任务目标需要做提炼筛选，保持与任务的相关性。
	- **执行编排**
		- 理解目标
		- 判断信息是否完整 -> 不够就补
		- 继续分析 -> 生成输出
		- 检查输出 -> 不满足就修正重试
	- **状态与记忆**
		- 当前任务状态
		- 会话中间结果
		- 长期记忆与偏好
	- **评估和观测**
		（评估生成结果的质量）
		- 输出验收
		- 环境验证
		- 自动测试
		- 日志和指标
		- 错误归因
	- **约束、校验与失败恢复**
		- 约束
		- 校验
		- 恢复

## 实现架构

### 三Agent架构

**架构组成**：

- **Planner** 扩展需求
- **Generator** 生成代码
- **Evaluator** 交互验证
	比如使用 Playwright 工具做交互式验证和打分。
	解决自评估的系统缺陷，Agent总是对自己生成的内容表现地很自信。

**三 Agent 架构产品案例**：

- **Aletheia**
- **ADK** （开源 Agent 开发套件）

## 功能模块

- **上下文工程**（Context Engineering）
	- **项目指令文件**
		AGENTS.md/ CLAUDE.md: 作为 Agent 启动时读取的知识基座。
	- **动态上下文注入**
		从日志、指标、追踪信息中获取实时信息管道。
	- **上下文隔离**
		子 Agent 作为上下文防火墙，隔离窗口独立运行。
	- **上下文压缩**
		窗口填满时自动丢弃或摘要无关信息，对抗熵增。

- **工具编排**
	- **精选工具集**
		工具不是越多越好，精心策划，移除冗余选项
	- **MCP协议集成**
	- **沙箱隔离**
		执行环境与生产系统隔离，如 Stripe devbox 模式

- **验证机制与硬约束**
	- **确定性约束**（不需要LLM）
		- 自定义 Linter
		- 结构测试
		- Pre-commit
	- **生成-评估分离**
		- 独立评估器
		- Anthropic 模式
	- **自动审查循环**
		- 多 Agent 审查
		- 迭代至满意
		传统开发中，人类工程师负责代码审查（Code Review）。在驾驭工程中，这个工作变成了"**智能体对智能体**"的方式：Codex 在本地审核自身更改，请求额外审查，循环往复直到通过。
		反馈循环中的钩子可以运行预定义的测试套件，并在失败时带着错误信息循环回到模型，或者提示模型独立评估其代码。如果 AI 写的测试用例"通过"了带有 Bug 的代码，Harness 就会判定测试无效，强迫它重新思考测试边界。

- **状态管理与记忆持续性**

- **可观测性**
	- 执行追踪
	- 质量分级
	- 异常检测
	- 反馈归因
		失败模式追溯到 Harness 缺陷，驱动持续改进。

- **人类接管与生命周期管理**
	- 关键节点暂停让人类确认是否继续

- **熵管理**（Entropy Management）——垃圾回收
	定期清理项目迭代过程中积累的技术债务。
	具体措施：定期运行后台 Codex 任务扫描偏差、更新质量等级、发起针对性重构 PR。此外还有一个专门的 **Doc-gardening Agent**（文档园丁代理），在后台自动扫描文档与代码之间的不一致，发现过时内容就自动提交 PR 修复——Agent 为 Agent 维护文档。
